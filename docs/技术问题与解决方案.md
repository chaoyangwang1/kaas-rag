# KAAS-RAG 关键技术问题与解决方案

---

## 问题一：多路检索结果质量参差不齐，简单拼接导致答案质量下降

### 问题描述

系统同时运行 4 路检索（向量搜索、HyDE、知识图谱、网络搜索），但不同检索源的结果质量差异巨大。对于简单技术参数查询，HyDE 可能生成不相关的假设文档导致检索偏离，网络搜索结果可能来自非权威来源。简单拼接所有结果送给 LLM 会导致答案质量下降甚至错误。

### 解决方案：RRF 加权融合 + BGE-Reranker 精排 + 动态 TopK 断崖截断（三阶段排序管道）

**阶段一：带权重的 RRF 倒数排名融合**

- 在 `node_rrf.py` 中实现带权重版本的 RRF 算法
- 核心公式：`score = Σ weight × 1/(k + rank)`
- 当前配置：k=60, Embedding 权重 1.0, HyDE 权重 1.0
- 不同检索源可通过 `source_weights` 参数赋予不同权重，让可靠来源获得更大话语权

**阶段二：BGE-Reranker 精细重排序**

- 将所有 RRF 融合结果 + Web 搜索结果合并，送入 BGE-Reranker-large 模型
- 使用 `(query, passage)` 配对计算语义相关性分数
- 按分数降序排列，实现从粗排到精排的质变

**阶段三：动态 TopK 断崖截断**

- 在 `node_rerank.py` 的 `step_3_topk_dynamic` 中实现智能截断
- 从 min_topk 条之后开始扫描相邻文档分数差值
- 当相对差值 ≥ 0.25（gap_ratio）或绝对差值 ≥ 0.5（gap_abs）时触发断崖截断
- 只保留断崖之前的高相关性文档，自动过滤低质内容

### 为什么这么做

1. **RRF 不依赖原始分数分布**：不同检索源的分数尺度不一致（余弦距离、BM25、KG 评分），RRF 只依赖排名而非分数绝对值，天然适合多源融合。
2. **三阶段排序是业界最佳实践**：粗排（检索）→ 融合（RRF）→ 精排（Reranker），逐级提纯，兼顾效率和精度。
3. **断崖截断利用分数自然分布**：Reranker 打分后，相关文档和不相关文档之间存在显著分数断崖，通过检测这个断崖自然截断，比固定 TopK 更智能。

## 问题二：用户口语化查询与产品手册术语不匹配，检索召回率低

### 问题描述

用户查询往往是口语化的（如"这个机器怎么开"），而产品手册中使用的术语是标准化的（如"设备启动步骤"）。直接使用原始查询检索会导致语义不匹配，召回率低下。此外，用户可能使用指代词（"这个""它"）引用上文提到但当前问题中未明确的主体。

### 解决方案：商品名确认 + 查询改写 + HyDE 双保险

**第一步：商品名确认（`node_item_name_confirm`，7步流程）**

1. 获取最近 10 条历史对话
2. 保存用户当前消息到 MongoDB
3. 使用 LLM (`rewritten_query_and_itemnames` prompt) 从对话中联合提取商品名和改写查询
4. 将商品名向量化后在 Milvus 的 `kaas_item_names` 集合中搜索标准化名称
5. 置信度对齐：>0.85 直接确认；0.6~0.85 作为候选让用户选择；<0.6 拒答
6. 分支决策：确定商品名→继续检索；多个候选→反问用户；无结果→提示拒答
7. 写入历史记录

**第二步：查询改写（`node_query_rewrite`）**

- 使用专门的 `query_rewrite.prompt` 模板，要求 LLM 将口语化问题转化为适合检索产品手册的规范化查询
- 优先使用确认的商品名增强查询语句

**第三步：HyDE 假设文档检索**

- 使用 LLM 基于改写后的查询生成一段假设性理想答案
- 将"原始查询 + 假设文档"拼接后向量化检索
- 利用 LLM 生成的"完整答案"与真实文档片段在语义空间更接近的特性

### 为什么这么做

1. **商品名确认解决指代消解问题**：在多轮对话中，用户说"它怎么维修"，系统需要知道"它"指代什么。通过维护历史对话 + LLM 提取，将指代词解析为标准化商品名。
2. **查询改写解决术语鸿沟**：口语→术语的转换让查询向量与文档向量在同一语义空间，直接提升余弦相似度。
3. **HyDE 的核心洞察**（Gao et al., 2022）：`P(d|q)` 在密集向量空间中，假设文档 `d_hat`（LLM 生成的答案）的向量比短查询 `q` 更接近真实文档 `d` 的向量分布。

## 问题三：Embedding 服务首次调用慢、重复计算浪费资源

### 问题描述

BGE-M3 模型加载耗时较长（首次需要数秒），每次查询都要重新生成相同的 query embedding（如热门问题被反复提问），造成不必要的延迟和计算资源浪费。此外，超长文本送往远程 Embedding 服务会触发 413 Payload Too Large 错误。

### 解决方案：单例模式 + LRU 缓存 + 文本截断 + 本地/远程双模式

**1. 模型单例加载 (`get_bge_m3_ef`)**

- 全局单例 `_bge_m3_ef`，首次使用后缓存，后续调用直接返回
- 支持本地模式（直接加载 FlagEmbedding）和远程模式（HTTP 调用 Docker 服务）

**2. LRU 进程内缓存**

- 在 `embedding_utils.py` 中实现进程内 LRU 缓存
- 键为 `MD5(text)`，值为 `{dense, sparse}` 字典
- 最大 1000 条，自动淘汰最早条目
- 日志中输出命中率统计

**3. 远程模式超长文本截断**

- 远程模式下自动将超过 500 字符的文本截断，防止 API 413 错误
- 本地模式不截断，充分利用 BGE-M3 的长文本能力

**4. 远程+本地双保险**

- 稠密向量从远程服务获取，稀疏向量优先从远程响应提取
- 远程未返回稀疏向量时，自动回退到本地 BGE-M3 模型生成

### 为什么这么做

1. **单例避免重复模型加载**：BGE-M3 模型约 2GB，每次加载需要 3-5 秒。单例模式只需要加载一次。
2. **LRU 缓存针对热门查询**：在实际场景中，用户查询存在长尾分布，20% 的热门 query 会产生 80% 的 embedding 调用，缓存可以覆盖这些热点。
3. **灵活的双模式满足不同部署场景**：开发环境用本地模式（无网络依赖），生产环境用远程 Docker 服务（GPU 加速、可水平扩展）。

## 问题四：大语言模型输出结构化数据不稳定，JSON 解析容易失败

### 问题描述

在商品名识别节点中，需要 LLM 输出结构化的 JSON（包含 `item_names` 和 `rewritten_query` 字段）。但 LLM 可能返回：
- 带 Markdown 代码块的 JSON（` ```json\n{...}\n``` `）
- 格式不规范的多余文本
- 缺少必需字段的 JSON

这导致后续的 `json.loads()` 解析失败，使整个检索流程不可用。

### 解决方案：JSON Mode + 健壮性解析 + Markdown 代码块清理

**1. JSON Mode 强制输出标准 JSON**

- 在 `lm_utils.py` 的 `get_llm_client()` 中，当 `json_mode=True` 时设置 `model_kwargs["response_format"] = {"type": "json_object"}`
- 这是 OpenAI 兼容 API 的原生功能，模型会强制返回合法的 JSON 对象
- 缓存键增加 `json_mode` 维度，不同模式的客户端独立缓存

**2. Markdown 代码块清理**

```python
if content.startswith("```json"):
    content = content.replace("```json", "").replace("```", "")
```

- 自动清除 ` ```json ` 和 ` ``` ` 标记
- 保留纯 JSON 字符串供 `json.loads()` 解析

**3. 字段健壮性检查**

```python
if "item_names" not in result:
    result["item_names"] = []
if "rewritten_query" not in result:
    result["rewritten_query"] = query
```

- 即使 JSON 不完整，也能用默认值兜底
- 保证后续检索流程不会因缺失字段而崩溃

### 为什么这么做

1. **JSON Mode 是模型的底层约束而非 prompt 工程**：在 prompt 中说"请返回 JSON"只是建议，模型仍可能违规。而 `response_format: json_object` 是在 token 采样层面对生成进行约束，从根本上保证输出格式。
2. **双重防御策略**：JSON Mode 解决了 90% 的情况，Markdown 清理和字段检查兜底剩下 10%。
3. **系统可靠性优先**：在 RAG 系统中，单一节点的 JSON 解析失败不应导致整个问答流程崩溃。通过默认值兜底，即使 LLM 输出异常，系统仍能以降级模式运行。

## 问题五：系统参数分散硬编码，调优困难，不同知识库需要不同检索策略

### 问题描述

系统的检索参数（chunk_size、top_k、RRF k 值、Rerank 阈值、各路搜索开关等）分散在 10+ 个文件中硬编码。针对不同类型的产品手册（技术手册 vs 操作指南 vs 安全手册），最优参数组合完全不同。若要为每个知识库定制检索策略，需要改动大量代码，无法灵活适配。

### 解决方案：知识库级可调参数体系 + MySQL 持久化 + API 动态配置

**1. MySQL 持久化配置**

在 `knowledge_bases` 表的 `config_json` 字段中存储 JSON 格式的完整参数配置，每次查询实时从数据库读取：

```json
{
  "import": {
    "chunk_max_length": 2000,
    "chunk_min_length": 500
  },
  "embedding_search": {
    "dense_weight": 0.8,
    "sparse_weight": 0.2,
    "req_limit": 10,
    "top_k": 5
  },
  "rrf": {
    "k": 60,
    "max_results": 10,
    "embedding_weight": 1.0,
    "hyde_weight": 1.0
  },
  "rerank": {
    "max_topk": 10,
    "min_topk": 1,
    "gap_ratio": 0.25,
    "gap_abs": 0.5
  },
  "features": {
    "enable_hyde": true,
    "enable_web_search": true,
    "enable_kg_search": true,
    "enable_query_rewrite": true
  },
  "generate": {
    "max_context_chars": 12000,
    "temperature": 0.1
  }
}
```

**2. 逐级参数覆盖链**

```
全局默认常量（RERANK_MAX_TOPK=10）
    ↓ 被覆盖
知识库配置（kb_config.rerank.max_topk=5）
    ↓ 被覆盖
请求级参数（POST /query 中的 enable_hyde=false）
```

每个节点从 `state["kb_config"]` 中读取参数，支持按知识库定制：
- `node_search_embedding.py`: 从 `kb_config.embedding_search` 读取 dense_weight/sparse_weight/req_limit/top_k
- `node_rrf.py`: 从 `kb_config.rrf` 读取 k/max_results/各路权重
- `node_rerank.py`: 从 `kb_config.rerank` 读取 max_topk/min_topk/gap_ratio/gap_abs
- `main_graph.py`: 从 `kb_config.features` 读取各路搜索开关，动态决定 Send API 分叉数量

**3. API 动态管理**

`/api/kb` 路由提供完整的 CRUD 接口，可通过 API 创建/更新/查询知识库配置，无需重启服务即可生效。

**4. 删除保护机制**

删除知识库前检查是否有已完成的文档，防止误删有价值的向量数据。

### 为什么这么做

1. **不同产品手册有不同检索需求**：技术参数手册（短句为主）适合关键词搜索 → 调高稀疏权重；操作指南（长文本描述）适合语义搜索 → 调高稠密权重。
2. **JSON 持久化比环境变量更灵活**：环境变量只能存储扁平的键值对，JSON 支持嵌套结构，可管理 30+ 个参数且清晰归类。
3. **MySQL 读取保证实时生效**：每次查询都从 MySQL 读取最新配置，不需要重启服务，支持前端界面动态调参。
4. **删除保护防止数据事故**：生产环境中误删知识库会导致大量向量数据丢失且不可恢复，保护机制是必要的安全兜底。

## 补充问题反思

这 5 个问题的选择覆盖了 RAG 系统全链路的关键环节：

| 问题 | 覆盖环节 | 核心挑战 |
|---|---|---|
| 多路检索融合 | 检索→排序 | 多源结果质量差异处理 |
| 查询语义匹配 | 意图理解→检索 | 口语化与术语的语义鸿沟 |
| Embedding 性能 | 基础设施 | 模型加载与计算效率 |
| LLM 结构化输出 | LLM 调用 | 模型输出的不确定性 |
| 可调参数体系 | 系统架构 | 多租户多场景适配 |
