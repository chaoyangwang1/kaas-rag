## 一、基于 LangGraph 的有状态工作流编排

### 亮点描述

系统采用 LangGraph 的 `StateGraph` 构建了两条完整的有状态工作流（知识库导入 + 智能问答），每条工作流通过 `TypedDict` 定义完整的状态结构，利用条件边（`add_conditional_edges`）实现动态分支路由。

### 核心优势

- **Send API 并行执行**：查询流程使用 LangGraph 的 `Send` API 实现 4 路搜索的真正并行执行。传统方案需要手动管理线程池，而 LangGraph 原生支持基于状态的并行分叉和自动合并，代码量减少 60%+
- **条件路由自然建模**：导入流程根据文件类型（PDF/MD）自动选择分支路径；查询流程根据意图确认结果自动路由到反问输出或继续检索
- **状态驱动**：所有节点共享一个状态对象，节点间数据传递通过字典更新实现，无需额外的消息队列或全局变量

### 代码示例（查询流程并行分叉）
```python
def fanout_to_searches(state: QueryGraphState):
    searches = [Send("node_search_embedding", state)]  # 始终开启
    if features.get("enable_hyde", True):
        searches.append(Send("node_search_embedding_hyde", state))
    if features.get("enable_web_search", True):
        searches.append(Send("node_web_search_mcp", state))
    if features.get("enable_kg_search", True):
        searches.append(Send("node_query_kg", state))
    return searches
```

## 二、BGE-M3 稠密+稀疏双向量混合搜索

### 亮点描述

系统使用 BGE-M3 模型同时生成稠密向量（1024维，语义匹配）和稀疏向量（词级匹配），在 Milvus 中执行加权混合搜索，兼顾语义理解和关键词匹配两种检索范式。

### 核心优势

- **互补检索**：稠密向量捕获语义相似性（"设备启动" ↔ "开机步骤"），稀疏向量保留词级匹配能力（型号精确匹配 "ECM-200B"）
- **加权融合**：通过 `WeightedRanker` 实现可调的稠密/稀疏权重比，默认 (0.8, 0.2)，可根据知识库类型定制
- **本地/远程双模式**：支持本地直接加载 BGE-M3 模型（FlagEmbedding）或调用 Docker 远程服务，适配不同部署需求
- **LRU 缓存**：进程内 LRU 缓存 Embedding 结果（最大 1000 条），热点查询直接命中，减少重复计算

### Milvus 混合搜索封装
```python
# 稠密向量用 COSINE 相似度，稀疏向量用 IP（内积）
dense_req = AnnSearchRequest(data=[dense_vector], anns_field="dense_vector",
                              param={"metric_type": "COSINE"}, limit=limit)
sparse_req = AnnSearchRequest(data=[sparse_vector], anns_field="sparse_vector",
                               param={"metric_type": "IP"}, limit=limit)
# WeightedRanker 加权融合
rerank = WeightedRanker(dense_weight, sparse_weight)
```

## 三、HyDE（Hypothetical Document Embedding）增强检索

### 亮点描述

在标准向量检索之外，系统引入 HyDE 技术——先让 LLM 为问题生成一段"假设性理想答案"，再用这段假设文档去做向量检索。

### 核心原理（Gao et al., 2022）

在密集向量空间中，LLM 生成的假设文档 `d_hat` 的向量分布更接近真实文档 `d`，而短查询 `q` 的向量往往与文档向量存在语义空间不匹配。通过"生成假设文档 → 向量化 → 检索"的路径，将短查询的语义稀疏问题转化为长文档的语义匹配问题。

### 实现细节

- `step_1_create_hyde_doc()`: 调用 LLM 基于改写后的查询生成假设性文档
- `step_2_search_embedding_hyde()`: 将"查询 + 假设文档"拼接后生成向量在 Milvus 中检索
- 支持知识库级配置开关：通过 `features.enable_hyde` 控制是否启用

## 四、七步商品名确认流程 —— 解决多轮对话中的指代消解

### 亮点描述

用户在多轮对话中常使用指代词（"这个怎么修"），系统需要准确识别用户指代的产品。为此设计了完整的 7 步确认流程：

| 步骤 | 功能 | 核心逻辑 |
|---|---|---|
| Step 1 | 获取历史 | MongoDB 查询最近 10 条对话 |
| Step 2 | 保存消息 | 初始保存用户消息到 MongoDB |
| Step 3 | LLM 联合提取 | 从用户问题+历史中提取商品名和改写查询 |
| Step 3.5 | 反问匹配 | 检测上一轮助手是否给出了选项列表，匹配用户选择 |
| Step 4 | 向量检索 | 在 Milvus item_names 集合中搜索标准名称 |
| Step 5 | 置信度对齐 | >0.85 确认 / 0.6~0.85 候选 / <0.6 拒答 |
| Step 6 | 分支决策 | 更新状态，决定后续路由 |
| Step 7 | 历史写入 | 更新用户消息中的 item_names |

### 智能反问匹配

当用户回复候选选项时（如"第一个"、"选2"、"HAK 180"），系统通过正则匹配 + 模糊包含自动解析用户意图，无需 LLM 再次调用，节省延迟。

## 七、动态 TopK 断崖截断算法

### 亮点描述

在 Reranker 打分后，传统方案使用固定 TopK（如取前 10 条）。本系统实现了基于分数断崖检测的动态 TopK：

### 算法逻辑

```
按分数降序排列后，从第 MIN_TOPK 条开始向后扫描：
  score_1=0.95  ← 始终保留
  score_2=0.91  ← 始终保留
  score_3=0.88  ← 从这条开始检查与下一条的差距
  score_4=0.42  ← gap=0.46 ≥ 0.5(绝对阈值) → 断崖！
  score_5=0.15  ← 丢弃
  ...
最终保留前 3 条（在断崖处截断）。
```

### 优势

- **自适应**：根据文档的实际相关性分布自动决定保留数量，而非固定值
- **双重阈值**：相对阈值（gap_ratio=0.25）+ 绝对阈值（gap_abs=0.5），覆盖不同分数范围
- **减少噪声**：自动过滤与问题不相关的文档，提升 LLM 答案质量
- **节省 Token**：减少送入 LLM 的上下文长度，降低成本和延迟

## 八、SSE 流式输出 + 实时进度追踪

### 亮点描述

系统实现了完整的 SSE（Server-Sent Events）流式推送机制，在查询过程中实时向前端推送：
- `progress` 事件：实时任务进度（正在执行的节点 + 已完成的节点）
- `delta` 事件：LLM 逐字生成的文本
- `final` 事件：最终完整答案 + 提取的图片 URL 列表 + 实体选项

### 核心实现

- **异步解耦**：后台线程执行查询任务 → 通过 `asyncio.Queue` 解耦 → SSE 异步推送到前端
- **节点级别进度**：30+ 个节点名→中文名映射，前端展示友好的进度信息
- **图片智能提取**：从 Markdown 正文中自动提取图片 URL，在 final 事件中传给前端渲染

## 十、多层次连接管理与容错机制

### 亮点描述

1. **MySQL 连接池**：线程本地存储（`threading.local()`），每个线程独立连接，避免并发冲突；自动 ping 重连
2. **Milvus 客户端单例**：全局单例复用连接，避免重复创建
3. **MinIO 客户端单例**：启动时自动创建 bucket 并配置只读策略；上传失败不阻塞主流程
4. **MongoDB 重试机制**：3 次重试 + 指数退避 + ping 验证，应对 Docker 容器启动时序问题
5. **LLM 客户端缓存**：按 `(模型名, JSON模式, api_key, base_url)` 四元组缓存 ChatOpenAI 实例
6. **Embedding 重试机制**：远程 HTTP 调用支持 3 次重试（1s→2s→4s 指数退避），应对偶发断连

## 十一、完整的检索评估体系

### 亮点描述

系统内置了完整的 RAG 检索评估模块，支持：
- **数据集管理**：创建/导入（JSON/CSV）/编辑/删除测试数据集
- **多引擎评估**：自定义评估引擎 + RAGAS 标准评估引擎
- **多组参数对比**：同一数据集多组参数并行评估，横向对比不同配置的检索效果
- **MongoDB 持久化**：数据集、条目、任务、结果全部持久化存储
- **后台线程执行**：评估任务异步执行，支持取消操作
