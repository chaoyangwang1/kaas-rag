# 系统可调参数梳理

> 所有硬编码参数的位置、含义、当前值和动态化方案。

---

## 一、导入流程参数

### 1.1 文档切分

| 参数 | 当前值 | 位置 | 含义 |
|---|---|---|---|
| `DEFAULT_MAX_CONTENT_LENGTH` | 2000 | `node_document_split.py:17` | 单个 chunk 最大字符数，超过触发二次切分 |
| `MIN_CONTENT_LENGTH` | 500 | `node_document_split.py:19` | 短 chunk 合并阈值，低于此值且同父标题则合并 |

**调整影响：**
- `MAX` 增大 → chunk 更完整但可能包含无关信息，LLM 上下文利用率下降
- `MAX` 减小 → chunk 更精准但可能切断关键句子，检索召回率下降
- `MIN` 增大 → 更激进合并，chunk 数更少，碎片化低但可能合并了不同语义段落
- `MIN` 减小 → 碎片更多，检索更精准但 LLM 需要处理更多 chunk

## 二、意图确认流程参数

### 2.1 置信度阈值

| 参数 | 当前值 | 位置 | 含义 |
|---|---|---|---|
| 高置信度阈值 | 0.85 | `node_item_name_confirm.py:181` | 高于此值直接确认为目标商品 |
| 中置信度阈值 | 0.6 | `node_item_name_confirm.py:182` | 0.6~0.85 之间作为候选让用户选择 |
| 低置信度阈值 | <0.6 | 隐式 | 低于 0.6 直接拒答 |

**调整影响：**
- 高阈值上调 → 更保守，更多反问，准确率高但用户体验下降
- 高阈值下调 → 更激进，更多自动确认，召回率高但可能选错商品

## 三、检索流程参数

### 3.1 向量搜索

| 参数 | 当前值 | 位置 | 含义 |
|---|---|---|---|
| `req_limit` | 10 | `node_search_embedding.py:73` | Milvus 底层候选召回数量（Embedding 路） |
| `req_limit` | 10 | `node_search_embedding_hyde.py:54` | Milvus 底层候选召回数量（HyDE 路） |
| `top_k` | 5 | `node_search_embedding.py:84` | 最终返回 Top K（Embedding 路） |
| `top_k` | 5 | `node_search_embedding_hyde.py:55` | 最终返回 Top K（HyDE 路） |
| `ranker_weights` | (0.8, 0.2) | `node_search_embedding.py:83` | 稠密/稀疏加权比例（Embedding 路） |
| `ranker_weights` | (0.8, 0.2) | `node_search_embedding_hyde.py:56` | 稠密/稀疏加权比例（HyDE 路） |

**调整影响：**
- `req_limit` 增大 → 更多候选进入后续流程，召回率高但 RRF 和 Rerank 计算量增大
- `top_k` 增大 → 单路贡献更多结果给 RRF，融合更充分
- `ranker_weights` 稠密偏大 → 偏语义匹配，适合自然语言描述性查询
- `ranker_weights` 稀疏偏大 → 偏关键词匹配，适合型号/编号查询

### 3.2 是否启用各路检索

| 参数 | 当前状态 | 位置 | 含义 |
|---|---|---|---|
| HyDE 检索 | 始终启用 | `main_graph.py` — fanout_to_searches 固定四路 | 通过 LLM 生成假设文档后再检索 |
| Web 搜索 | 始终启用 | `main_graph.py` — fanout_to_searches 固定四路 | 联网搜索补充外部信息 |
| 知识图谱 | 始终启用 | `main_graph.py` — fanout_to_searches 固定四路 | Neo4j 查询实体关系 |

**当前问题：** 不管用户问什么，四路全开。70% 的简单 query 其实向量检索就够了。

**动态化方案：** 在意图确认节点增加一路判断——根据 query 类型选择要启用的搜索路数：
- 简单技术参数 → 仅向量搜索
- 操作步骤 → 向量 + HyDE
- 需要外部信息 → 向量 + Web 搜索
- 复杂多实体关系 → 全部四路

## 四、RRF 融合参数

| 参数 | 当前值 | 位置 | 含义 |
|---|---|---|---|
| `k` | 60 | `node_rrf.py:179` | RRF 平滑常数，控制排名差异的平滑程度 |
| `max_results` | 10 | `node_rrf.py:179` | RRF 融合后保留的文档数量 |
| Embedding 路权重 | 1.0 | `node_rrf.py:173` | 在 RRF 分数中的加权系数 |
| HyDE 路权重 | 1.0 | `node_rrf.py:174` | 在 RRF 分数中的加权系数 |
| 知识图谱路权重 | 未参与 | `node_rrf.py` | 当前 KG 结果未接入 RRF |
| Web 搜索路权重 | 未参与 | `node_rrf.py` | 当前 Web 结果未接入 RRF |

**调整影响：**
- `k` 增大 → 排名差异更平滑，头部优势被稀释，适合多源结果质量差异大的场景
- `k` 减小 → 头部排名更有话语权，适合"头部文档通常正确"的场景
- 各路权重不同 → 可以让某一路的排名在融合中占更大比重（如知识图谱高精度可给更大权重）

**注意：** 知识图谱和 Web 搜索结果当前**未接入 RRF**，而是直接跳到 Rerank 的 `step_1_merge_docs` 中与 RRF 结果合并。这意味着它们不参与排名融合，只在 Rerank 阶段竞争。

## 五、Rerank + 动态 TopK 参数

| 参数 | 当前值 | 位置 | 含义 |
|---|---|---|---|
| `RERANK_MAX_TOPK` | 10 | `node_rerank.py:10` | 重排序后最多保留的文档数 |
| `RERANK_MIN_TOPK` | 1 | `node_rerank.py:12` | 重排序后最少保留的文档数 |
| `RERANK_GAP_RATIO` | 0.25 | `node_rerank.py:14` | 断崖检测的相对阈值（相邻分数降幅比例） |
| `RERANK_GAP_ABS` | 0.5 | `node_rerank.py:16` | 断崖检测的绝对阈值（相邻分数绝对差值） |

### 断崖截断逻辑

```
按分数降序排列后，从第 MIN_TOPK 条开始向后扫描：

  score_1=0.95  ← 始终保留
  score_2=0.91  ← 始终保留
  score_3=0.88  ← 从这条开始检查与下一条的差距
  score_4=0.42  ← gap=0.46 ≥ 0.5(绝对阈值) → 断崖！截断在 index=3
  score_5=0.15
  ...

最终保留前 3 条。
```

**调整影响：**
- `GAP_RATIO` 增大 → 更宽容，更多文档通过，但可能包含不相关文档
- `GAP_RATIO` 减小 → 更严格，更早截断，但可能漏掉相关文档
- `GAP_ABS` 同上，但基于绝对分值差
- `MAX_TOPK` 是安全上限，防止一次送给 LLM 过多文档导致 token 溢出

## 九、参数关联关系（调参时需要注意）

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  文档切分                                                        │
│  MAX_CONTENT_LENGTH=2000 ──────┐                                 │
│                                 │                                │
│  向量检索                       ▼                                │
│  req_limit=10 ──────────────→ RRF 融合                           │
│  top_k=5 ──────────────────→  k=60, max_results=10               │
│  ranker_weights=(0.8,0.2) ─→  source_weights=(1.0, 1.0)         │
│                                 │                                │
│  HyDE检索                       │                                │
│  req_limit=10, top_k=5 ──────→ │                                │
│                                 │                                │
│  KG + Web（未接入RRF）          ▼                                │
│  ──────────────────────────→ Rerank + 动态TopK                   │
│                               MAX_TOPK=10, MIN_TOPK=1            │
│                               GAP_RATIO=0.25, GAP_ABS=0.5        │
│                                 │                                │
│                                 ▼                                │
│                          答案生成                                │
│                          MAX_CONTEXT_CHARS=12000                 │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**关键依赖链：**
1. `chunk_size` 小了 → 需要更多 chunk 才能覆盖完整答案 → `req_limit` 和 `top_k` 可能需要相应调大
2. `req_limit` 和 `top_k` 调大 → RRF `max_results` 也需要调大，否则提前截断浪费了检索的 recall
3. Rerank `MAX_TOPK` 调大 → `MAX_CONTEXT_CHARS` 不够用，可能被截断 → 需要联动调整
4. HyDE/Web/KG 关了 → RRF 只剩一路，`max_results` 可以直接等于 `top_k`，k 值也不需要平滑多路

## 十、动态化优先级建议

| 优先级 | 参数 | 理由 |
|---|---|---|
| P0 | 是否启用 HyDE 检索 | 每次查询省 1.5s，简单 query 不需要 |
| P0 | 是否启用 Web 搜索 | 省外部 API 调用，离线场景可关闭 |
| P1 | `ranker_weights` | 不同 query 类型最优权重不同 |
| P1 | RRF `k` 值 | 不同场景效果差异大 |
| P1 | `top_k` / `req_limit` | 检索精度 vs 延迟的权衡 |
| P2 | Rerank 断崖阈值 | 微调 TopK 质量 |
| P2 | `MAX_CONTENT_CHARS` | 适配不同 LLM 上下文窗口 |
| P3 | chunk_size | 低频调整，需重新导入 |

### 动态化实现方案

**方案 A：环境变量（.env）** — 适合低频调整的参数

```bash
# .env
RAG_CHUNK_MAX_LENGTH=2000
RAG_RRF_K=60
RAG_RERANK_GAP_RATIO=0.25
RAG_SEARCH_DENSE_WEIGHT=0.8
RAG_ENABLE_HYDE=true
RAG_ENABLE_WEB_SEARCH=true
```

在 `app/conf/` 下新增 `rag_config.py`，统一加载。

**方案 B：请求级参数** — 适合需要按 query 动态调整的参数

```python
# 前端请求中传入
POST /query
{
    "query": "ECM-200B 电压多少",
    "enable_hyde": false,      # 简单查询跳过 HyDE
    "enable_web_search": false,
    "top_k": 3                 # 自定义召回数量
}
```

**方案 C：自动推理** — 系统根据 query 特征自动选择

在意图确认节点中增加判断逻辑：
- query 包含型号/编号 → 调高稀疏权重
- query 纯自然语言 → 调高稠密权重
- query 很短（<10 字） → 跳过 HyDE
- 知识库内置信度高 → 跳过 Web 搜索
